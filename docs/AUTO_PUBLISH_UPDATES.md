# Auto-Publishing Updates From Your Commits

## In Plain English (start here)

You wanted updates to publish themselves when you push. Here's how it works now,
in one sentence: **when you write a commit, add a line that starts with `Update:`
(for the public page) or `Dev-Update:` (for the engineering page), and the app
turns that line into a real entry automatically.**

- `Update:` lines become entries on the public **/updates** page (the plain-English
  one athletes see). For safety they appear as a **draft** first — hidden from the
  public until you flip one switch. That stops anything embarrassing or half-baked
  from going live on its own.
- `Dev-Update:` lines become entries on the **/dev-updates** page (the engineering
  log) and go **live right away**, because that page is meant to be technical.
- Commits **without** one of those lines are ignored. So your normal "fix typo",
  "refactor", or "bump dependency" commits never clutter the public page.

Nothing is ever pushed for you — you still push when you're ready, exactly like
today. The entries simply ride along with your push and appear after the site
redeploys.

**One-time setup:** run this once in a terminal so the magic happens on every commit:

```
npm run hooks:install
```

That's the whole idea. The rest of this doc is the detail.

---

## The two trailers

A "trailer" is just a line at the **end** of your commit message, like `Key: value`.
You only ever need these two:

```
Update: You can now record your swing right inside the app.
Dev-Update: Added in-app MediaRecorder capture with a framing overlay.
```

A commit can have one, both, or neither. Example commit message:

```
feat(video): in-app swing recording

Adds front/back camera capture with a "where to stand" overlay.

Update: You can now record your swing right inside the app — no separate camera app needed.
Dev-Update: Browser MediaRecorder capture with a framing guide, behind an opt-in flag.
```

> Tip: keep each trailer on a **single line**. If you want a longer description,
> use the optional fields below.

## Turning it on (one time per computer)

```
npm run hooks:install
```

This tells git to run the project's `post-commit` hook. From then on, every time
you commit, the app:

1. reads your `Update:` / `Dev-Update:` lines,
2. writes the entries into the two data files, and
3. makes a tiny follow-up commit (`chore(updates): publish entries from commit trailers`)
   containing **only** those data files.

When you push, the entries deploy with everything else.

To turn it off again: `git config --unset core.hooksPath`.

You can also run it by hand any time, without the hook:

```
npm run updates:generate
```

## Publishing a product (/updates) draft

New `Update:` entries are created as `"status": "draft"` and `"visibility": "private"`,
so they do **not** show on the public page yet. When you're happy with the wording:

1. Open `apps/web/src/data/auto-updates.json`
2. Find your entry (it has `"autoGenerated": true` and a `"sourceCommit"` matching your commit)
3. Change `"status": "draft"` → `"status": "published"`
4. Change `"visibility": "private"` → `"visibility": "public"`
5. Commit and push. It goes live on the next deploy.

(Developer `/dev-updates` entries have no draft step — they're live as soon as they deploy.)

## Optional fields (only if you want more control)

All optional, all single-line. Skip any you don't need — sensible defaults fill in.

### Product (`Update:`) extras

| Trailer | What it does | Default |
|---|---|---|
| `Update-Title:` | A shorter headline, if your `Update:` line is long | the `Update:` line |
| `Update-Summary:` | The 1–2 sentence summary | the `Update:` line |
| `Update-Benefit:` | The "What this means for you" line | the summary |
| `Update-Why:` | The "Why it matters" line | (omitted) |
| `Update-Where:` | The "Where to find it" line | (omitted) |
| `Update-Action:` | A short "Action" callout | (omitted) |
| `Update-Category:` | One of the valid categories (see below) | `Product Updates` |
| `Update-Sport:` | `Golf`, `Tennis`, `Baseball`, `Slow Pitch Softball`, `Fast Pitch Softball`, or `All Sports` | (none) |
| `Update-Audience:` | Comma-separated, e.g. `parents, coaches` | `all athletes` |

Valid categories: `New Feature`, `Training Improvement`, `Equipment`, `Data & Insights`,
`Multi-Sport Expansion`, `Golf Training`, `Tennis Training`, `Baseball Training`,
`Softball Training`, `Video & Swing Comparison`, `Progress Tracking`, `Account & Data`,
`Mobile Experience`, `Website`, `SEO & Discoverability`, `Security & Privacy`, `Product Updates`.

### Developer (`Dev-Update:`) extras

| Trailer | What it does | Default |
|---|---|---|
| `Dev-Title:` | A shorter title | the `Dev-Update:` line |
| `Dev-Details:` | A short technical paragraph | the `Dev-Update:` line |
| `Dev-Category:` | `AI & Vision`, `Motion Intelligence`, `Architecture`, `Platform`, `Performance`, `Design System`, `Security & Privacy`, `Developer Experience` | `Platform` |
| `Dev-Impact:` | `major`, `notable`, or `foundational` | `notable` |
| `Dev-Stack:` | Comma-separated tech tags, e.g. `TypeScript, Next.js` | (none) |
| `Dev-Highlights:` | Semicolon-separated bullet points | (none) |
| `Dev-Milestone:` | `true` to pin it on the engineering timeline | `false` |

A fuller example:

```
feat(motion): kinetic chain timing

Update: Motion Lab now shows how your power flows from the ground up.
Update-Category: Training Improvement
Update-Sport: All Sports
Update-Where: Open Motion Lab, analyze a swing, and check the Scores tab.

Dev-Update: Kinetic-chain engine times when each link peaks and flags power leaks.
Dev-Category: Motion Intelligence
Dev-Impact: major
Dev-Stack: TypeScript, Canvas 2D, Jest
Dev-Highlights: Depth-aware sequencing; power-leak detection; per-phase durations
```

## Good to know

- **No duplicates.** Each entry is keyed by its commit's short SHA, so running the
  generator twice never creates a second copy.
- **Nothing is auto-pushed.** The hook only commits the two data files locally
  (with an explicit file list — it never sweeps up other work). You push manually.
- **It can't block a commit.** If anything goes wrong in the hook, your commit
  still succeeds; the entry just isn't generated.
- **Quality bar still applies.** The plain-English writing rules in
  [HOW_TO_PUBLISH_UPDATES.md](HOW_TO_PUBLISH_UPDATES.md) are the standard for what a
  good `Update:` line looks like. Write it for an athlete, not a developer.

## Files involved

| File | Role |
|---|---|
| `scripts/generate-updates.mjs` | The generator (reads commits → writes entries) |
| `scripts/hooks/post-commit` | Runs the generator after each commit |
| `scripts/install-hooks.mjs` | One-time setup (`npm run hooks:install`) |
| `apps/web/src/data/auto-updates.json` | Product `/updates` entries (drafts land here) |
| `apps/web/src/data/auto-dev-updates.json` | Developer `/dev-updates` entries (live) |
